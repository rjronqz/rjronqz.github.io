Backend to Infrastructure

A study route for a backend developer who has never provisioned a server. Seven phases, each with the concepts to learn, sources to read, one hands-on lab, and a checkpoint. You are done with a phase when you can pass its checkpoint without notes.

ESTIMATED: 12–16 weeks part-time PREREQ: any backend language COST: ~$5–6/mo for one VPS (phases 3+)
Map 1 · Linux 2 · Networking 3 · First server 4 · Repeatability 5 · Docker 6 · Kubernetes Appendix Beyond
Orientation

The map

Everything in this manual serves one storyline: take an API you wrote, and move it from "runs on my laptop" to "runs replicated, self-healing, behind a load balancer." Each phase removes one layer of magic. Kubernetes is deliberately last — it only makes sense once you have felt the pain it solves.

Ground ruleDo the labs. Reading about infrastructure produces familiarity; breaking and repairing a real server produces skill. Every phase's lab builds on the previous one, using the same small API you already have (or a 20-line hello-world HTTP service).
Phase 1

Linux: the machine itself

2–3 weeks · local, free

Every server you will ever touch — VPS, container, Kubernetes node — is a Linux box. The terminal is the interface for all of it. This phase is about being at home there: knowing where things live, who is allowed to touch them, and what is running.

Exit skillYou can SSH into an unfamiliar Linux machine and, without a search engine, find its logs, running services, disk usage, and open ports.

Concepts

Topic What to actually learn
Shell fluency Navigation, pipes and redirection (|, >, 2>&1), grep, find, less, tail -f, environment variables, $PATH.
Filesystem hierarchy What belongs in /etc, /var/log, /usr/bin, /opt, /home, /tmp, /proc. Why config and data live apart from binaries.
Users & permissions Users, groups, rwx bits, chmod/chown, sudo, why services run as unprivileged users.
Processes ps, top/htop, PIDs, signals (SIGTERM vs SIGKILL), exit codes, foreground vs background.
systemd Units, systemctl start/enable/status, writing a basic .service file, reading logs with journalctl -u.
Packages apt update vs apt upgrade, where packages install files, repositories.

Sources

  • Linux Journey — free, sequenced lessons; do "Grasshopper" and "Journeyman" tracks.
  • The Linux Command Line (free book) — chapters 1–10 and 24–26 are the highest-value reading here.
  • OverTheWire: Bandit — shell practice as a game; levels 0–20 cement the basics.
  • systemd.service man page — reference for the lab; skim, don't memorize.
  • Julia Evans' comics — one-page visual refreshers on processes, signals, permissions.

Lab 1 — a supervised service, locally

Deliverable: your API running as a systemd service in a local VM
  1. Install Multipass (or VirtualBox) and launch an Ubuntu VM. This is your disposable practice server.
  2. Create a non-root user app; copy a small HTTP service you wrote into /opt/myapp, owned by that user.
  3. Write /etc/systemd/system/myapp.service that runs it as app, with Restart=on-failure.
  4. systemctl enable --now myapp, then curl localhost:8080 from inside the VM.
  5. Kill the process with kill -9 and watch systemd resurrect it (journalctl -u myapp -f).
  6. Reboot the VM and confirm the service comes back on its own.
Checkpoint — answer without notes
  • Where do you look first when a service "isn't working" and why?
  • What's the difference between SIGTERM and SIGKILL, and why does it matter for a database?
  • Why should your API not run as root?
Phase 2

Networking: how bytes travel

2–3 weeks · local, free

As a backend developer you live at the top of the network stack. This phase fills in what happens beneath your framework: how a hostname becomes an IP, how a TCP connection is a real stateful thing, and what a port actually is. Almost every production incident you'll ever debug touches this layer.

Exit skillGiven "the site is down," you can methodically test DNS, connectivity, TLS, and the app itself — in that order — using only a terminal.

Concepts

Topic What to actually learn
The layered model TCP/IP layers as separation of concerns: link → IP (addressing/routing) → TCP/UDP (ports, reliability) → application (HTTP). Don't memorize OSI trivia; learn what each layer is responsible for.
IP & CIDR IPv4 addresses, private ranges (10.x, 192.168.x), subnet masks, what /24 means, NAT, why your laptop's IP isn't public.
Ports & sockets A socket = IP + port + protocol. Listening vs connecting. ss -tlnp to see who's listening. Why two processes can't bind the same port.
DNS Resolution chain (root → TLD → authoritative), record types (A, AAAA, CNAME, TXT, MX), TTL and caching, dig.
HTTP & TLS Raw HTTP/1.1 over a socket, status codes, headers; the TLS handshake at a high level; what a certificate proves; curl -v.
Firewalls Default-deny thinking; ufw; the difference between "connection refused" and "connection timed out."

Sources

  • Cloudflare Learning Center — clear explainers: "What is DNS", "What is TCP/IP", "What happens in a TLS handshake".
  • Linux Journey — Networking Nomad track — subnetting, routing, DNS from the Linux side.
  • Julia Evans — networking zine — the fastest way to build intuition for the tools below.
  • Mess With DNS — a sandbox domain to experiment with real DNS records safely.
  • MDN — HTTP overview — authoritative reference for the protocol you already use daily.

Lab 2 — trace a request by hand

Deliverable: a written trace of one HTTPS request, produced with CLI tools
  1. Pick any site. Resolve it step by step: dig example.com, then dig +trace example.com. Identify the A record and TTL.
  2. Speak raw HTTP: printf 'GET / HTTP/1.1\r\nHost: example.com\r\n\r\n' | nc example.com 80. Read the response headers.
  3. Inspect TLS: curl -v https://example.com — find the handshake, the certificate issuer, and expiry.
  4. In your Phase-1 VM: ss -tlnp to list listeners; stop your service and watch its port disappear.
  5. Enable ufw in the VM, block port 8080, and observe how the failure mode changes (refused vs timeout).
Checkpoint — answer without notes
  • A user gets your app's old IP after you migrated servers. What DNS concept explains it and what should you have done beforehand?
  • "Connection refused" vs "connection timed out" — what does each suggest?
  • Why can nginx talk plain HTTP to your app on 127.0.0.1 without undermining HTTPS?
Phase 3

Your first real server

2–3 weeks · ~$5–6/mo VPS

Now you rent a real machine on the public internet and deploy the old-fashioned way: SSH in, harden it, put nginx in front, get a TLS certificate, wire up systemd. This is deliberately manual. Docker and Kubernetes automate exactly these steps — you can't appreciate the automation until you've done it by hand once.

Exit skillYour API is live at https://yourdomain.com on a server you set up and secured yourself, surviving reboots.

Concepts

Topic What to actually learn
Provisioning What a VPS is (a VM on someone's hardware); choosing size; SSH key auth at creation time. Hetzner and DigitalOcean are the common starter choices.
SSH properly Key pairs, ~/.ssh/config, ssh-agent, disabling password login and root login in sshd_config.
Hardening basics ufw default-deny, fail2ban, unattended security upgrades, principle of least privilege for the app user.
Reverse proxy Why nginx sits in front of app servers: TLS termination, buffering, static files, one IP → many apps, graceful restarts.
TLS certificates Let's Encrypt + certbot; what the ACME challenge proves; auto-renewal.
Deploying Getting code on the box (git pull / rsync / scp), environment variables for secrets, running the DB locally, backups with pg_dump + cron.
Housekeeping Time sync: verify timedatectl shows "NTP service: active" and clock synchronized — Ubuntu ships systemd-timesyncd by default; install chrony if you need tighter sync. Drift silently breaks TLS validation, JWT/TOTP expiry, and cron. Keep the server timezone UTC. Also: logrotate and journald size limits so logs can't fill the disk, and df -h as a habit — full disks are the #1 boring outage. For kernel/process limits (ulimits, sysctl) see Appendix A — read it once now, apply it only when a symptom appears.
High-risk moments in this phase (1) Editing sshd_config or enabling ufw can lock you out permanently. Always allow SSH (ufw allow OpenSSH) before ufw enable, and keep your current SSH session open while testing a new one in a second terminal. (2) Disable password auth only after confirming key login works. (3) On a fresh VPS, bots start probing port 22 within minutes — do the hardening steps first, not "later." (4) Note for the future: the ufw rules you write here protect host processes only — Docker will later bypass them entirely. See the "host vs container" caveat in Phase 5 before you install Docker on this box.

Sources

  • DigitalOcean — Initial Server Setup with Ubuntu — the canonical walkthrough; DO's community tutorials are the best in this space regardless of which host you pick.
  • nginx Beginner's Guide + DO's "nginx server blocks" and "nginx reverse proxy" tutorials.
  • Certbot instructions — pick your OS + nginx; it configures TLS for you.
  • SSH Academy — reference for keys, agent, config.
  • LandChad.net — opinionated, terse guides for exactly this "own your first server" phase.

Lab 3 — go live

Deliverable: your API public at https://api.<yourdomain>.com
  1. Buy a cheap domain; create a VPS with your SSH public key preloaded.
  2. Point an A record at the VPS IP. Verify with dig before moving on.
  3. Harden: new sudo user, key-only SSH, ufw allowing only 22/80/443, fail2ban, unattended-upgrades. Then timedatectl set-timezone UTC and confirm NTP is active and synchronized.
  4. Install your runtime + Postgres; repeat Lab 1's systemd setup for your API on 127.0.0.1:8080.
  5. Configure nginx as a reverse proxy for your domain → localhost:8080; then run certbot for HTTPS.
  6. Break it on purpose: stop the app (502 from nginx), stop nginx (connection refused), remove the DNS record (resolution failure). Recognize each signature from the outside.
  7. Set a nightly pg_dump cron job and restore it once to prove the backup is real.
Checkpoint — answer without notes
  • Walk through every hop of FIG.2 for your own domain, naming the real config file or service at each step.
  • Why does your API bind to 127.0.0.1 and not 0.0.0.0?
  • What exactly does Let's Encrypt verify before issuing your certificate?
Phase 4

Repeatability: scripts over snowflakes

1–2 weeks · same VPS

Your phase-3 server is now a snowflake: unique, hand-crafted, and unreproducible. If it dies, your knowledge dies with it. The whole modern infra stack — Docker, Kubernetes, Terraform — exists to answer one question: "how do I get this exact setup again, on demand?" This short phase plants that mindset.

Exit skillYou can destroy your VPS and have an identical replacement serving traffic within 15 minutes, from code.

Concepts

Topic What to actually learn
Bash scripting Variables, conditionals, set -euo pipefail, exit codes — enough to automate your Lab-3 steps.
Idempotency The key idea: scripts that describe a desired state and can run repeatedly, vs scripts that replay actions and break the second time.
Config management Ansible at an introductory level: inventory, playbooks, modules. It's "SSH in and do the steps," codified and idempotent.
Infrastructure as Code Terraform/OpenTofu at awareness level only: declaring the VPS itself (and DNS records) in code. Deep-dive later, not now.
12-factor apps Config in env vars, logs to stdout, stateless processes — the app-side contract that makes phases 5–6 possible.

Sources

  • The Twelve-Factor App — short, foundational; read all of it.
  • Ansible — Getting Started — official docs are genuinely good; stop after "playbooks."
  • Fowler — Snowflake Server (and Phoenix Server) — the two-page argument behind this whole phase.
  • Terraform — Intro — read for concepts; hands-on can wait until after Kubernetes.

Lab 4 — the phoenix test

Deliverable: an Ansible playbook that rebuilds your production server from zero
  1. Write a playbook that performs every Lab-3 step: user, hardening, packages, app deploy, systemd unit, nginx config, certbot.
  2. Run it against a fresh second VPS. Fix everything you forgot (there will be things).
  3. Run it twice — the second run should report no changes. That's idempotency.
  4. The phoenix test: destroy the original VPS, re-point DNS, and rebuild with one command. Time yourself.
Checkpoint — answer without notes
  • What makes a script idempotent, and why does that property matter for infra tools?
  • Which 12-factor rules did your app violate before this phase?
Phase 5

Docker: ship the environment

2–3 weeks · local + VPS

Ansible reproduced your server's setup; containers go further and package the app with its environment, so the same artifact runs identically on your laptop, your VPS, and (soon) a Kubernetes node. A container is not a VM — it's an isolated process on the host kernel, which is why it starts in milliseconds.

Exit skillYou can containerize any of your backend apps with a lean multi-stage Dockerfile and run app + DB with one docker compose up.

Concepts

Topic What to actually learn
Image vs container Image = immutable template (layers); container = a running instance. Analogy from your world: class vs object.
Dockerfile FROM, COPY, RUN, CMD vs ENTRYPOINT; layer caching and why instruction order matters; multi-stage builds for small images; .dockerignore.
Registries Docker Hub / GHCR; tags; push/pull; why latest is a trap in production.
Networking Port publishing (-p 8080:8080), user-defined bridge networks, containers resolving each other by name.
State Container filesystems are ephemeral; volumes and bind mounts; where your Postgres data must live.
Compose Declaring a multi-container stack in compose.yaml: services, networks, volumes, env, depends_on, healthchecks.
Host vs container — things that stop working the way Phase 3 taught you
  • Docker bypasses ufw. ufw filters the INPUT chain, but Docker writes its own iptables rules (in the FORWARD/NAT path) for published ports — so -p 5432:5432 exposes Postgres to the internet even if ufw denies it. Handle it by: publishing to loopback only (-p 127.0.0.1:5432:5432) for anything nginx or the host should reach privately; putting internet-facing filtering in the DOCKER-USER chain (the hook Docker provides for your rules); and/or using your cloud provider's external firewall, which Docker can't touch. Read Docker — packet filtering and firewalls before exposing anything. Do not set "iptables": false — it breaks container networking.
  • localhost changes meaning. Inside a container, 127.0.0.1 is the container itself, not the host. Host-nginx → container: proxy to the published loopback port. Container → host service: use the docker bridge gateway (or host.docker.internal where supported). Container → container: use the compose service name, never localhost.
  • No systemd inside. Your process is PID 1 in the container: supervision moves to restart policies (restart: unless-stopped) or the orchestrator, and PID 1 must handle SIGTERM itself or it gets SIGKILLed after the grace period. Use the exec form of CMD (CMD ["./server"], not shell form) so signals actually reach your process; add init: true if it spawns children.
  • Logs move. journalctl shows nothing for containers — logs are whatever the process writes to stdout/stderr, read via docker logs. This also means fail2ban-style tools that tail host log files can't see containerized services without extra wiring.
  • The clock belongs to the host. Containers share the host kernel's clock — never run NTP inside a container; keep chrony/timesyncd healthy on the host and every container is correct for free. (Timezone is separate: containers default to UTC regardless of the host's TZ — fine, keep it that way and convert at the edges.)

Sources

  • Docker — Get Started + Docker Workshop — official, hands-on, current; do it end to end.
  • Dockerfile best practices — read after your first working Dockerfile, then rewrite it.
  • Docker Compose docs — the services/volumes/networks model you'll reuse mentally in Kubernetes.
  • Docker — Packet filtering and firewalls — required reading before running containers on an internet-facing host; explains the ufw bypass and DOCKER-USER.
  • Play with Docker — free in-browser hosts for experiments.

Lab 5 — containerize the stack

Deliverable: your app + Postgres running via compose, image on a registry, deployed to the VPS
  1. Write a Dockerfile for your API. Get it working, then convert to a multi-stage build and compare image sizes.
  2. Write compose.yaml: your app + postgres with a named volume; the app reaches the DB at hostname db, config via env vars.
  3. Prove persistence: docker compose down (data survives) vs down -v (data gone). Understand why.
  4. Tag and push the image to GHCR or Docker Hub.
  5. On your VPS: install Docker, docker compose up -d your stack, keep nginx+TLS in front. Retire the systemd unit from Lab 3.
  6. Verify the ufw bypass yourself: temporarily publish Postgres as -p 5432:5432, confirm with nmap (or nc) from your laptop that it's reachable despite ufw — then fix it with 127.0.0.1:5432:5432 and re-test. Never skip the re-test.
Checkpoint — answer without notes
  • Why can a published container port be reachable from the internet even though ufw denies it, and what are two correct fixes?
  • Why does moving COPY . . below dependency installation make builds faster?
  • Your container writes uploads to its own filesystem. What happens on redeploy, and what's the fix?
  • What does a container share with its host that a VM doesn't?
Phase 6

Kubernetes: orchestrating many

4+ weeks · local (kind/minikube), free

One VPS running compose is fine until you need: more than one machine, zero-downtime deploys, automatic restarts on node failure, or scaling under load. Kubernetes is a control loop over a fleet: you declare desired state in YAML, and controllers work continuously to make reality match. That declarative model is the single idea everything else hangs off.

Exit skillYou can deploy your compose stack to a local cluster with Deployments, Services and Ingress, then kill pods and nodes and explain exactly how it heals.

Concepts — learn in this order

Layer What to actually learn
The model Declarative desired state; reconciliation loops; kubectl apply / get / describe / logs / exec. describe + Events is your new journalctl.
Pods The scheduling unit: one or more containers sharing network + storage. Pods are ephemeral by design — never precious.
Deployments ReplicaSets, rolling updates, rollbacks. The object you'll write most.
Services Stable name + virtual IP over ever-changing pods; ClusterIP vs NodePort vs LoadBalancer; cluster DNS.
Ingress HTTP routing into the cluster — nginx again, now as an Ingress controller. Your Phase-3 knowledge maps directly.
Config & Secrets ConfigMaps and Secrets injected as env vars or files; same 12-factor contract as compose.
Health & resources Liveness/readiness probes; CPU/memory requests and limits; what OOMKilled means.
State (later) PersistentVolumes/Claims, StatefulSets, Namespaces, Helm. Skim now, deepen after the lab.
Host vs cluster — the firewall story, continued
  • The node's iptables belongs to Kubernetes now. kube-proxy programs iptables (or IPVS) on every node to implement Services, and the CNI plugin wires pod networking on top. Never hand-edit iptables or run ufw with your Phase-3 rules on a node — you'll either be silently overridden or break pod/Service traffic in ways that are miserable to debug.
  • NodePort Services open ports on every node (default range 30000–32767), on all interfaces, regardless of what you think your host firewall allows. On real clusters, the perimeter belongs to the cloud security group / external firewall in front of the nodes — not to per-node rules.
  • Inside the cluster, the firewall is a Kubernetes object. By default every pod can reach every other pod. Segmentation is done with NetworkPolicy — declarative, label-based rules ("only pods labeled app=api may reach the db on 5432"). Caveat inside the caveat: policies are only enforced if the CNI supports it (Calico, Cilium do; kind's default kindnet does not — create your kind cluster with Calico if you want the lab drill below to actually block anything).
  • fail2ban/sshd hardening still applies to nodes as machines, but app-level protection moves up the stack: Ingress controller rate limits, NetworkPolicies, and admission policies replace the host-centric tools from Phase 3.
  • Node housekeeping still matters — and two items are cluster-critical. (1) Clock skew: every control-plane↔kubelet connection is mutual-TLS, and etcd is a distributed consensus store — drifting node clocks cause certificate-validation failures, misordered events, and unexplainable "works on one node" bugs. Run chrony on every node (managed clouds do this for you; your own nodes are your problem). (2) Swap: kubelet refuses to start with swap enabled by default (swapoff -a + remove from fstab) — a classic first-cluster blocker on self-managed nodes.

  • Kubernetes docs — Concepts + Kubernetes Basics tutorial — the primary source; read Concepts in the table's order.
  • kind (or minikube) — real multi-node clusters in Docker on your laptop; free and disposable.
  • Killercoda K8s scenarios — free browser labs; good daily reps.
  • Kubernetes the Hard Way — build a cluster from raw components — once, after you're comfortable, to destroy the remaining magic.
  • CKA/CKAD certification paths — optional, but the CKAD syllabus is a solid practical curriculum even if you never sit the exam.

Lab 6 — compose → cluster

Deliverable: your Phase-5 stack on a local multi-node cluster, demonstrably self-healing
  1. Create a 3-node kind cluster. Explore: kubectl get nodes, kubectl get pods -A (see the system itself running as pods).
  2. Write a Deployment (3 replicas of your image) + ClusterIP Service. kubectl port-forward to hit it.
  3. Move env config into a ConfigMap and DB credentials into a Secret.
  4. Add Postgres via a StatefulSet with a PersistentVolumeClaim (accept some copy-adapting from docs — that's normal).
  5. Install the ingress-nginx controller and route http://localhost/api to your Service.
  6. Chaos drills: kubectl delete pod (watch replacement), roll out a bad image tag then rollout undo, drain a node and watch pods reschedule, set a memory limit low enough to see OOMKilled.
  7. Add readiness + liveness probes; redo the bad-image drill and note how rollout behavior changes.
  8. Firewall drill: recreate the cluster with Calico as the CNI, apply a default-deny NetworkPolicy in your namespace, watch the app break, then allow only api→db:5432 and ingress→api. Verify with kubectl exec + nc from an unrelated pod.
Cost warning for laterWhen you eventually try managed Kubernetes (EKS/GKE/AKS), control planes and load balancers bill by the hour. Create, experiment, and tear down the same day — a forgotten cluster is a $70+ surprise. Local kind clusters are free; stay local until you have a reason not to be.

  • What's the difference between a readiness probe failing and a liveness probe failing?
  • Where does traffic filtering live at each layer of a real cluster: internet→nodes, pod→pod, and node SSH access?
Appendix A

ulimits & sysctl, by server role

Kernel limits exist to protect a general-purpose machine from a runaway process. Dedicated servers routinely need them raised — but tuning is symptom-driven, not ritual. Every row below names the symptom the value fixes; if you don't have the symptom, keep the default. Whatever you change, record it in your Phase-4 playbook (drop files in /etc/sysctl.d/ and /etc/security/limits.d/), never one-off in a live shell.

Where limits actually live — three traps
  • /etc/security/limits.conf does not affect systemd services. It's applied by PAM at login. Your API from Lab 1 gets its file-descriptor limit from LimitNOFILE= in its unit file (or systemd's defaults), not from limits.conf. Test what a process actually got: cat /proc/<pid>/limits.
  • Containers don't read the host's limits.conf either. They inherit ulimits from the container runtime (dockerd default-ulimits, or --ulimit / compose ulimits: per container). Most net.* sysctls are per-network-namespace; most fs.*/vm.*/conntrack sysctls are host-wide — you tune those on the node, and in Kubernetes only a short "safe" list is settable per-pod via securityContext.sysctls.
  • Verify, then trust: sysctl <key> reads live values; sysctl --system reapplies your drop-in files; ulimit -n only tells you about your current shell.

Baseline — any internet-facing app server

Setting Guidance & why
LimitNOFILE 65535 in the service's unit file. Every TCP connection, file, and pipe is a file descriptor; the historical soft default of 1024 dies at ~1k concurrent connections. Symptom: EMFILE / "too many open files" in app logs.
net.core.somaxconn The cap on any listener's accept backlog (kernel default 4096 since 5.4; older kernels 128). Raise to 1024–4096 together with the app's own backlog setting (nginx listen ... backlog=, defaults to 511) — the effective value is the minimum of the two. Symptom: connection resets/timeouts during traffic bursts while CPU is idle; overflow counters in netstat -s | grep -i listen.
net.ipv4.ip_local_port_range Widen to 1024 65535 only on boxes making heavy outbound connections (proxies, scrapers, service-to-service callers). Symptom: EADDRNOTAVAIL / "cannot assign requested address".
net.ipv4.tcp_tw_reuse 1 helps the same outbound-heavy case (safely reuses TIME_WAIT ports for new outgoing connections). Never touch its infamous sibling tcp_tw_recycle — it broke clients behind NAT so badly it was removed from the kernel in 4.12; any guide still recommending it is your signal the guide is stale.
vm.swappiness Default 60 is tuned for desktops. 10 is a common server value: prefer dropping page cache over swapping process memory. Symptom: latency spikes with si/so activity in vmstat while RAM isn't actually exhausted.

Database hosts

Setting Guidance & why
vm.overcommit_memory A teaching contrast, straight from each vendor's docs: Redis wants 1 (always overcommit) because background saves fork() the whole process — with strict accounting the fork fails ("Cannot allocate memory") even though copy-on-write means the child barely uses any. PostgreSQL suggests 2 (strict, with vm.overcommit_ratio) so the kernel returns clean allocation errors instead of letting the OOM killer assassinate the postmaster mid-transaction. Same knob, opposite values, each justified by how the engine uses memory — the reason cargo-culting sysctls across roles fails.
Transparent Huge Pages Disable (never) for Redis and MongoDB per their docs: khugepaged's background page-merging causes multi-ms latency stalls and bloats fork copy-on-write. PostgreSQL: THP off, but explicit huge pages on (vm.nr_hugepages + huge_pages=on) for large shared_buffers — pre-allocated at boot, none of the defrag stalls.
vm.max_map_count 262144 — Elasticsearch/OpenSearch refuse to bootstrap below this (mmap-heavy index files). Harmless for others; the classic "ES won't start" fix.
vm.dirty_background_ratio / vm.dirty_ratio On big-RAM boxes the defaults (10/20% of RAM) let tens of GB of dirty pages pile up, then flush in one I/O storm. Lowering (e.g. 3/10, or absolute *_bytes values) trades throughput for smoothness. Symptom: periodic multi-second stalls where every query hangs at once.
ulimits LimitNOFILE=65535 again (ES bootstrap-checks it), plus LimitMEMLOCK=infinity if you use ES bootstrap.memory_lock or Postgres huge pages. And per Phase 3: swap effectively off or heavily discouraged on dedicated DB hosts.

Docker / Kubernetes nodes

Setting Guidance & why
net.ipv4.ip_forward 1 — a node is a router for its pods/containers; traffic must cross from the bridge/CNI interfaces to the outside. Docker sets it for you; kubeadm's preflight check fails without it.
br_netfilter + net.bridge.bridge-nf-call-iptables=1 Makes bridged (pod-to-pod) traffic traverse iptables so kube-proxy's Service rules and NetworkPolicies can see it. The classic kubeadm setup step; without it, Services half-work in maddening ways.
fs.inotify.max_user_watches / max_user_instances 524288 / 512 — every kubelet, log tailer, and hot-reloading dev tool holds inotify watches, and dozens of containers multiply it. Straight from kind's known-issues page, where the symptom is pods failing with "too many open files."
net.netfilter.nf_conntrack_max Every connection through a node's NAT is a conntrack entry. Note who owns it: kube-proxy sets this itself (default 32768 × cores) — don't fight it with your own drop-in; on plain Docker hosts doing heavy traffic, raise it yourself (see next table for sizing).
ulimits Container fd limits come from the runtime, not the host's limits.conf (see traps box above). If one containerized app needs more, set it per-service in compose or the pod spec — not by raising the daemon default for everything.

Tunnel / NAT / SNAT gateways (VPN endpoints, egress boxes)

Setting Guidance & why
net.ipv4.ip_forward=1 The sysctl that makes the box a router. Without it, tunnels come up, handshakes succeed, and no traffic flows through — the most common WireGuard-gateway "bug."
net.netfilter.nf_conntrack_max On a NAT box, every flow from every client is a conntrack entry, so this is the first ceiling you hit. Size to RAM (~300 bytes/entry; 1M entries ≈ 300MB) and raise the hash table (hashsize) with it. Symptom is unambiguous: nf_conntrack: table full, dropping packet in dmesg — random connection drops for everyone behind the box.
nf_conntrack_tcp_timeout_established Default is 5 days (432000s) — a busy gateway accumulates dead flows for days. Dropping to 1–2 hours keeps the table honest. Don't go below your longest legitimate idle connection (long-polling, DB pools) or the keepalive interval of what's behind you.
SNAT port math Not a sysctl, a law: one public IP gives ~64k source ports per destination IP:port. Thousands of clients hitting the same big API through one gateway IP will exhaust it. Widening ip_local_port_range buys margin; the real fix is more SNAT IPs. Symptom: connection failures that only affect popular destinations.
net.ipv4.conf.all.rp_filter Strict reverse-path filtering (1) silently drops packets whose return route differs from their arrival route — normal life on multi-homed VPN gateways. Set 2 (loose) on tunnel boxes with asymmetric routing. Symptom: traffic visible in tcpdump arriving but never forwarded, no error anywhere.
MTU / MSS clamping Tunnels shrink the usable packet size (WireGuard's default MTU is 1420 for this reason). If path-MTU discovery is blackholed, you get the signature bug: ping and small requests work, large transfers hang forever. Fix with correct tunnel MTU and an iptables MSS-clamp rule (--clamp-mss-to-pmtu).
